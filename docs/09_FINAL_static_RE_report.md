## 2. Static extraction of the 4 JNI method pointers

The Java class `com.xingin.shield.http.Native` exposes 4 native methods:

```java
public static native void     destroy(long cPtr);
public static native long     initialize(String token);
public static native void     initializeNative();
public static native Response intercept(Interceptor.Chain chain, long cPtr);
```

None appear as `Java_com_xingin_shield_http_Native_*` symbols — they're all registered at runtime via `JNIEnv->RegisterNatives` from `JNI_OnLoad`. The `JNINativeMethod[4]` array is constructed on the stack inside JNI_OnLoad. ## 3. Anti-RE techniques used by libxyass

Every category of anti-RE was found in use:

### 3.1 Dynamic JNI registration
As shown above — no `Java_*` export symbols; `JNIEnv->RegisterNatives` called from `JNI_OnLoad` with a stack-built method table.

### 3.2 String obfuscation via lazy XOR decryption
Every runtime string (class names, method names, signatures, header names, error messages) is stored encrypted and decrypted on first access. The pattern inside JNI_OnLoad:

```c
if (init_flag_bit == 0) {                     // lazy init
    buf = malloc(len);                        // FUN_0001d7a4 allocator
    memcpy(buf, encrypted_bytes, len);        // copy ciphertext
    (*(code *)(fn_table[i] + big_const))(buf, len);  // call decryptor
    *cached_slot = buf;
}
use(*cached_slot);
```

- The decryption function pointer is **never stored plainly** — it's `stored_value + 0xXXXXXXXX` where the big constant is a compile-time obfuscation offset (different constants observed: `0x666e4b10`, `0x5886bbe4`, `0x72ac4290`, `0x3a1c95a8`).
- The decrypted strings are only visible after emulation runs them through the real decrypt routine. My attempt to brute-force the decrypt of `"destroy\0"` from the inline-immediate 8-byte blob `96 47 4a e8 93 b2 51 62` produced XOR key `f2 22 39 9c e1 dd 28 62` — **not** a constant, incrementing, or 4-byte-repeating pattern. Stream-cipher or APK-hash-keyed.

### 3.3 APK signing-certificate tamper check

Unicorn-traced the JNI lookup sequence (see `docs/07_libxyass_jni_surface.md`):

```
FindClass  android/app/ActivityThread
  GetStaticMethodID  currentApplication ()Landroid/app/Application;
  Call  ActivityThread.currentApplication() → Application
FindClass  android/app/Application
  GetMethodID  getPackageManager ()Landroid/content/pm/PackageManager;
  GetMethodID  getPackageName   ()Ljava/lang/String;
  Call  Application.getPackageManager()
  Call  Application.getPackageName()
FindClass  android/content/pm/PackageManager
  GetMethodID  getPackageInfo   (Ljava/lang/String;I)Landroid/content/pm/PackageInfo;
  Call  PackageManager.getPackageInfo(pkgName, GET_SIGNATURES)
FindClass  android/content/pm/PackageInfo
  GetFieldID   signatures       [Landroid/content/pm/Signature;
  GetField     PackageInfo.signatures
FindClass  android/content/pm/Signature
  GetMethodID  hashCode ()I
  → Call  signatures[0].hashCode() → int
```

The resulting `int` is used as the decryption key seed for the rest of JNI_OnLoad's string decryption. Computed Java `Signature.hashCode()` = `Arrays.hashCode(rawBytes)` for the XHS signing cert `META-INF/XINGIN.RSA` (942 B):

```
Java Arrays.hashCode(XINGIN.RSA) = 0x4cdc059d  (signed: +1289487773)
sha1:   648282ca90244fdb198371ee0ec619f5edf5147a
sha256: 74c209fc43b4c046306ce78bf293cc96cad3053dac5cb7c38ee44fad60e05160
```

(Note: Android's `Signature.mSignature` is actually the SignerInfo-level raw cert bytes, not necessarily the entire `.RSA` file — may need adjustment when running.)

### 3.4 Control-flow flattening in helper functions

Every helper function in the 0x1e000-0x1ec00 range (where JNI_OnLoad's string-decryption helpers live) uses register-based dispatch:

```
01ec02 cff6c541   movt  r1, #0xfcc5       ; r1 = base_low | 0xfcc5_0000
01ec06 8944       add   sb, r1             ; sb += r1
01ec08 8746       mov   pc, r0             ; jump to r0
01ec0a 43f23840   movw  r0, #0x3438       ; r0 = low
01ec0e cff63d40   movt  r0, #0xfc3d       ; r0 = 0xfc3d3438
01ec12 4844       add   r0, sb             ; r0 = 0xfc3d3438 + sb
01ec14 8746       mov   pc, r0             ; jump to r0 + sb (next dispatcher)
```

Each block ends with `mov pc, rX` where `rX = big_const + sb`. `sb` is an accumulator seeded at the start of the function and incremented by a per-block delta. Without the correct seed, jumps land outside `.text` and the emulator crashes. In real execution, `sb` is fed by values derived from the APK cert hash (from §3.3). My Unicorn emulation reached this dispatcher with wrong `sb` and halted.

### 3.5 NEON-based memory/crypto operations

Instructions like `vmov.i32 q8, #0`, `vst1.32 {d16, d17}, [r0]`, and `vpush {d8-d13}` are used for:
- Bulk zero-ing of state structures (80-byte state alloc at JNI_OnLoad + 0x12)
- Unknown further operations inside helper functions at `0x18930`+

NEON-heavy code was the first Unicorn roadblock — the Cortex-A15 VFP state apparently isn't fully enabled by Unicorn even with `uc.ctl_set_cpu_model(CORTEX_A15)` + `CPACR` write. Instructions at `0x18930` appear as `invalid` under Unicorn, despite being legitimate `LDMIA` / `VCVT.F16.U16` / `VTBL.8`. This is a known Unicorn / QEMU-arm limitation.

**Implications of this class list** (what we know the signer can and does do, just from this enumeration):

1. **`SharedPreferences` + `Editor`** — persistent per-install state is read and written. This is almost certainly the **per-device secret key `k`** and session identifiers `s`, `s1`, `gid` from the `x-mini-mua` payload (memory ID 1270). First install generates them; subsequent runs reuse them.
2. **`PackageManager` + `PackageInfo` + `Signature`** — APK tamper check. If you re-sign the APK, the hash changes, the keys it seeds all change, and the server will reject your signed requests.
3. **`okio/Buffer`** — the code buffers the **entire request body into memory** before signing (rules out streaming body signatures). Used with `RequestBody.writeTo(Buffer)` → `Buffer.readByteArray()` → hash input.
4. **`HttpUrl`** — signing depends on the full URL (host + path + sorted query string).
5. **`Headers`** — existing headers are read (probably for `Content-Type`, `Content-Length`, custom `X-Xhs-*` headers).
6. **`Request$Builder`** — the modified request (with added signature headers) is returned from `intercept()`.
7. **`Charset`** — UTF-8 encoding of the canonical string before hashing.
8. **`ContextHolder.writeLog(I)V`** — a Java-side logger is invoked when integrity check fails.

### 4.5 What intercept() does NOT call

- No `java.util.Random` / `SecureRandom` lookups — so the nonce bytes in the 100-B shield trailer are generated via a **custom PRNG** (either `/dev/urandom` via `syscall`, or a counter derived from `time()`).
- No `MessageDigest` / `Mac` / `Cipher` Java class lookups — so the crypto is **100 % native**, no Java crypto fallback.
- No reflection (`Method.invoke` / `Field.get`) — which confirms that the APK integrity check uses **direct JNI field access** (which is what we observed).

## 5. What remains unknown (and why)

For the canonical-string-builder and the actual hash function, pure static analysis is blocked by the following chain of obfuscations:

| Obstacle | Why it blocks static RE |
|---|---|
| **Every header name is XOR-encrypted** | You can't grep for `"x-mini-sig"` in `.rodata` — the string `"x-mini-sig"` only exists in decrypted heap memory after the decryptor runs. And the decryptor is gated on the APK cert hash. |
| **Decryptor is gated on APK cert hash** | Seed value has to be computed from a specific 942-byte RSA certificate blob. My best guess (`Arrays.hashCode(XINGIN.RSA)` = `0x4cdc059d`) may or may not be the right hash — Android's `Signature.mSignature` is technically a different byte sequence than the full `.RSA` file. |
| **CFG-flattened helpers** | Helper functions (including the decrypt routine) end each basic block with `mov pc, rX` where `rX = base + accumulator`. Wrong seed → jump target in `.dynstr` → Unicorn crash. |
| **NEON instructions at `0x18930`** | Unicorn mis-decodes legitimate Cortex-A15 NEON instructions (confirmed: `LDMIA r2!, {r3}` reported as "invalid"). Would need a different emulator (angr, or fs-qemu). |
| **MD5 T-table reachable only via obfuscated base+delta** | The T-table exists at `0x79418` but every MD5 constant has **zero immediate references** in `.text`. That means MD5 is accessed as `*(base + offset)` where `base` is seeded at init time — again gated on the APK-cert-derived decryption. |

**Net result**: even with a complete Ghidra + Capstone + Unicorn pipeline, recovering the algorithm statically means simultaneously solving the CFG flattening, the string decryption, and the Cortex-A15 VFP emulation issue. That is a **multi-day task** unsuitable for this session's time budget.

## 6. Algorithm model — evidence-supported vs. inferred

### 6.1 What is definitely true (from static evidence only)

- The signer is the native method `intercept(okhttp3.Chain, long cPtr)` @ `libxyass.so + 0x23e54`.
- It calls 8 `malloc()` and 3 different internal helpers (one called 13×, one 4×, one 3×).
- It uses `Request` / `HttpUrl` / `Headers` / `RequestBody` / `okio.Buffer` via JNI for **input** and `Request$Builder` for **output**.
- It reads / writes `SharedPreferences` on a per-install basis.
- It performs an APK signing certificate integrity check that must succeed for string decryption to work.
- No SHA-256, SM3, BLAKE2, AES, HMAC or other public crypto constants are present anywhere in `.text`. Only MD5 constants are present.
- `intercept()` does not import anything other than libc.

### 6.2 What is well-supported inference (but not 100 % confirmed statically)

- The three signature headers (`shield`, `x-mini-mua` trailer, `x-mini-sig`) are almost certainly derived from the same canonical request string — because otherwise there would be three independent buffer-building sequences in a 2682-byte function, but we see only one dominant `malloc(large)` + 13×`0x1edf8` helper pattern. One canonical string, three keyed hashes.
- The hash itself is **either custom MD5-based or HMAC-MD5 concatenated twice** (`MD5(k1,m) || MD5(k2,m)` → 32 bytes = the `x-mini-sig` length). No SHA-256 constants were found; MD5 T-table present; `x-mini-sig` is exactly 32 bytes / 64 hex.
- The per-device key `k` and session keys `s`, `s1`, `gid` live in `SharedPreferences` (persisted between runs) — consistent with `com.xingin.shield.http.ContextHolder` containing a `SharedPreferences` reference via JNI.

### 6.3 What was inference in earlier documents but is NOT supported by the static evidence

Document [docs/03_signature_algorithm_reverse_engineering.md](./03_signature_algorithm_reverse_engineering.md) §5 claimed the hashes were HMAC-SHA-256. **This was wrong.** `libxyass` has **no SHA-256** anywhere. Honest update: the hash is almost certainly MD5-based (or a custom construction that reuses the MD5 T-table).

## 7. Concrete next steps to close the remaining gap

Ranked by likelihood of completing the algorithm recovery:

### Option A (1–3 hours) — Frida dynamic hook on a real device
Already have the Frida script written at [scratch/ghidra_work/frida_dump_shield.js](../scratch/ghidra_work/frida_dump_shield.js). Modify it to:

## 8. Python port skeleton (what can be written today)

Based on what we know for sure, a Python implementation can be structured as:

```python
# scratch/ghidra_work/xhs_sign_skeleton.py  (committed, see same dir)

class XhsSigner:
    def __init__(self, shared_prefs_blob: bytes, apk_cert: bytes):
        # Parses the SharedPreferences dump from /data/data/com.xingin.xhs/shared_prefs/
        # Recovers: mua_k, mua_s, install_uuid, app_version, device_a_tag
        # These are the fields visible in decoded x-mini-mua JSON (known good).
        ...

    def sign(self, method: str, url: str, headers: dict, body: bytes) -> dict:
        canon = self._canonicalize(method, url, body)
        k_device   = self._load_device_key()          # from shared_prefs
        k_session  = self._load_session_key()         # from shared_prefs
        k_mua      = self._load_mua_key()             # from mua.k field

        # TODO: replace `custom_hash` with the actual hash once extracted via Frida.
        # Current best guess: MD5(key || canon) || MD5(key || canon || pad)
        # or HMAC-MD5 doubled.
        shield_blob = b'\x5d\x80\x00\x40' + self._custom_hash(k_device, canon) + self._nonce(48)
        mua_blob    = self._mua_json().encode() + self._custom_hash(k_mua, canon)[:32] + self._nonce(68)

        return {
            'shield'      : base64(shield_blob),
            'x-mini-mua'  : base64(mua_blob),
            'x-mini-sig'  : self._custom_hash(k_session, canon).hex()[:64],
            'x-mini-s1'   : self._cached_s1,     # static per session
            'x-mini-gid'  : self._cached_gid,    # static per session
            'x-legacy-did': self._install_uuid,
            'x-legacy-sid': self._session_id,
            'x-legacy-fid': '',
            # tracing headers — NOT signed
            'x-b3-traceid': random_hex(16),
            'x-xray-traceid': random_hex(32),
        }
```

This skeleton is complete for the output shape. The `_canonicalize()` format and `_custom_hash()` implementation are the two remaining black boxes that Option A unblocks in ~2 hours.

## 9. Replayability (already known to work)

Per memory records `1276` / `1278` / `1282` / `1292`: the XHS server **accepts replayed** `shield` / `x-mini-*` headers as long as the canonical string they sign hasn't changed (same method, path, sorted query, body). Many endpoints (`/api/sns/homefeed` and most read-only feed endpoints) don't require signatures at all — the `ux8.a0` interceptor strips headers before those requests go out.

For a practical scraper this means:
- **Easy path**: use unsigned endpoints directly (no shield at all). This is what memory entries 1278/1282 confirmed works.
- **Medium path**: capture one valid signed request per unique `(method, path, body)` tuple via mitm, and replay indefinitely.
- **Hard path** (this report's ultimate goal): fully reimplement the signer in Python. Needs Option A (Frida) to finish.
